# Syntactic ambiguity is recorded as spread confidence, never dropped

The syntactic pass resolves names by heuristic (ADR 0002), and a name can match several declarations. When it does, the pass emits an edge to every candidate and splits the confidence among them — N candidates get 0.9/N each (the constant `SYNTACTIC_MATCH`, one definition in the pass) — rather than picking a winner or dropping the reference. A consumer filtering `CALLS` at confidence ≥ 0.9 sees only unique matches; one accepting any confidence sees the full candidate set, ambiguity made legible instead of silently resolved. 0.9 is the ceiling for any *resolved* name: 1.0 is reserved for edges witnessed verbatim in the source — a DEFINES declaration, or the File → Package half of an IMPORTS edge (the import path is spelled right there) — where the only syntactic thing is the pass itself. The File → File half of an IMPORTS edge, by contrast, is a resolution (which repo files is this import path's package?) and so carries `SYNTACTIC_MATCH`, not 1.0; a blank or dot import (`import _ "…"`, `import . "…"`) is still a witnessed import and still mints its Package node and 1.0 edge, even though it introduces no name a call site could qualify with.

Resolution scope mirrors Go's: unqualified names resolve within the referencing file's package first (directory ≈ package) and fall back repo-wide only when the package has no candidate; package-qualified names (`util.Reset`) resolve through the file's imports via go.mod module paths and never fan out further; method calls (`x.Render()`) consider every method of that name repo-wide, since the receiver's type is invisible syntactically. An import path is mapped to repo directories by the *most specific* (longest) matching `module` path, so a nested go.mod owns its subtree and the parent module never also claims it. Names the repo doesn't declare — stdlib, external modules, builtins — yield no edge at all: the graph never invents targets. IMPLEMENTS is the one exception to the 0.9 ceiling: it matches method *sets by name only* (signatures are invisible to a reasonable syntactic pass), so it is capped at 0.5 (`NAME_ONLY_MATCH`) however unique the match, and interfaces with no directly declared methods (`any`) match nothing.

Edges that have a witnessed site carry a `location` of `<path>:<line>:<column>` (1-based; the column is a byte offset within the line, as tree-sitter reports it). The column is load-bearing, not cosmetic: two calls on one line are two distinct edges, and without the column they would be byte-identical rows. It is a byte offset rather than a display column deliberately — that is what makes it deterministic and collision-free for disambiguation; a consumer that wants an editor column maps bytes→characters against the source. The format packs into one string because line and column are always-present numeric suffixes, so a consumer recovers the path unambiguously by splitting off the last two colon-separated fields even when the path itself contains a colon.

Rejected: best-candidate-wins (silently wrong on collisions, and the error is invisible downstream); dropping ambiguous references (a graph that goes quiet exactly where the code is hardest to navigate — issue #6 explicitly requires the opposite). The precise pass (M1) overlays SCIP-resolved edges at provenance `precise`, which is the durable fix; spread confidence is how the syntactic floor stays honest until then.
